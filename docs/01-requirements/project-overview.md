# プロジェクト概要

## 背景・目的

### 背景

社内のプロジェクト管理業務は、担当者ごとにExcelや個別ファイルで管理されており、工数・品質情報が分散・属人化している状態にある。その結果、以下の課題が生じている。

- プロジェクト進捗・工数の全体把握に時間がかかる
- レビュー指摘や障害の管理が一元化されておらず、追跡が困難
- 実績データに基づく定量的な品質分析（EVM・信頼性成長曲線等）ができていない

### 目的

プロジェクトの進捗・工数・品質情報を一元管理するWebアプリケーションを社内構築し、データに基づいた意思決定を支援する。プロセスを標準化・見える化することで、プロジェクト管理の属人性を排除し、組織全体の開発品質向上を図る。

### 期待される効果

| 効果区分 | 内容 |
|---------|------|
| 工数管理の効率化 | 日報からの工数自動集計により、手作業での集計工数を削減 |
| 品質の見える化 | EVM・信頼性成長曲線・傾向分析によりリスクを早期検知 |
| プロセス標準化 | WBS・レビュー・障害管理の記録フォーマットを統一 |

## スコープ定義

### スコープ内

- **対象ユーザー**: 社内のプロジェクトマネージャー・開発者（〜20名規模）
- **対象環境**: PCブラウザ（デスクトップ画面）
- **対象機能**:
    - WBS・工数管理（タスク登録、予定/実績工数管理、ガントチャート表示）
    - 日報・実績入力（作業時間記録、バリデーション）
    - レビュー記録票管理（指摘登録・ステータス管理）
    - 障害管理票（バグトラッキング・WBSタスク紐付け）
    - レポート・分析（EVM、信頼性成長曲線、傾向分析）

### スコープ外

- モバイルアプリ（iOS/Android）への対応
- 外部システム連携（Jira、Slack、GitHubなど）
- 多言語対応（英語等）
- BIツール連携・外部レポーティングサービスへのデータ連携

### 前提条件・制約条件

| 区分 | 項目 | 内容 |
|------|------|------|
| 技術前提 | コンテナ基盤 | Dockerコンテナ上での稼働を前提とする |
| 技術前提 | フロントエンド | Next.js + Tailwind CSS + shadcn/ui |
| 技術前提 | データベース | PostgreSQL |
| 技術制約 | インフラ | 稼働インフラは未決定。Dockerを採用することで将来的なインフラ選択肢（オンプレ・クラウド）を確保する |
| 組織制約 | 利用規模 | 最大20名程度の小規模利用を前提とする |
| 組織制約 | 外部委託 | 社内開発チームのみで開発・運用する |

## ステークホルダー

| ステークホルダー | 役割 | 主な関心事 |
|----------------|------|-----------|
| 開発チームリーダー | プロジェクトオーナー・意思決定者 | 開発進捗・品質の全体把握、リスク早期検知 |
| プロジェクトマネージャー | 主要利用者 | 工数・進捗管理の効率化、EVM分析 |
| 開発者 | 主要利用者 | 日報入力・タスク管理の簡便さ |
| システム管理者 | 運用・保守担当 | 安定稼働、データバックアップ |

## 用語定義

| 用語 | 定義 |
|------|------|
| WBS | Work Breakdown Structure。プロジェクトのタスクを階層的に分解した構造 |
| EVM | Earned Value Management。PV（計画価値）・AC（実コスト）・EV（出来高）の3指標で進捗と費用効率を測る管理手法 |
| 予定工数 | WBSタスクに対して事前に見積もった作業時間（手動入力） |
| 実績工数 | 日報から自動集計される実際の作業時間 |
| 信頼性成長曲線 | 累積バグ発見数と累積修正数の推移を可視化し、ソフトウェアの品質収束状況を把握するグラフ |
| 指摘区分 | レビューで発見された指摘の分類（例：設計漏れ、実装誤り、テスト不足など） |
| 障害密度 | 担当者別・モジュール別の障害件数を規模で正規化した品質指標 |
